# 信息精炼

> 将原始内容精炼为结构化知识条目的标准化流程。
> 由 **Writer** 角色执行，作为信息挖掘管道中「置信度评估 → 格式化质检」之间的中间环节。

---

## 1. 触发条件与调用

| 项目 | 内容 |
|------|------|
| **触发条件** | 内容已通过置信度评估（> 60%），由 Coordinator 自动路由触发 |
| **执行角色** | Writer |
| **调用来源** | `SKILL/信息挖掘.md` §2.3 置信度评估 — > 60% 自动触发 |
| **自动触发链路** | Researcher 完成置信度评估 → Coordinator 判断 > 60% → 自动路由给 Writer → Writer 收到原始内容 + 置信度评分 + 评估理由 → 执行精炼六步法 |
| **输出目标** | 精炼完成后 Writer 自动按模板格式化，以 `YYYY-MM-DD_SourceName_Title.md` 命名保存到 `wiki/` |

---

## 2. 输入检查

开始精炼前，确认输入内容包含以下要素：

- [ ] **原始正文**：可读的 Markdown 文本
- [ ] **来源标签**：`[来源:XXX]` 格式（来自 Source Adapter）
- [ ] **置信度评分**：Researcher 已给出 `[置信度:0.XX]`
- [ ] **元数据初判**：至少包含主题/分类/tags 的初步判断

**缺少以上任一项** → 回退到 Researcher 补充，不跳过缺失项自行脑补。

---

## 3. 精炼六步法

### Step 1: 通读理解

通读全文，标记以下要素：
- **核心命题**：作者/嘉宾的核心观点
- **关键数据**：数字、统计、比例
- **背景信息**：上下文、前提、假设
- **冲突观点**：同一议题的不同立场

> 对于长内容（> 3000 字），先按自然段落分段，再逐段精炼。

---

### Step 2: 提取核心结论

从原文中提取 **3-5 个核心命题**。每个命题应满足：

| 标准 | 说明 |
|------|------|
| ✅ 独立可验证 | 有来源支撑，不是推测 |
| ✅ 信息密度高 | 不是过渡句、开场白、客套话 |
| ✅ 有知识增量 | 不是常识性内容 |
| ❌ 不提取 | 个人感受、模糊断言、无来源推测 |

**格式**：

```markdown
## 核心结论

1. **命题简述** [置信度:0.XX] [来源:XXX]
   → 展开说明（1-2 句）
```

---

### Step 3: 结构化组织

按以下三层结构重新组织内容：

```markdown
## 核心结论
（3-5 条精炼命题，每条附带置信度和来源）

## 详细内容
（按逻辑分组展开，保留重要上下文和论证过程）

## 来源
（标准化来源引用，附原文链接）
```

**结构原则**：
- 核心结论 = 精华版（可单独阅读）
- 详细内容 = 完整版（保留论证过程）
- 来源 = 追溯版（方便验真）

---

### Step 4: 知识标记

填写 YAML 头部元数据：

```yaml
---
type: knowledge          # knowledge / founder / insight
入库日期: YYYY-MM-DD
置信度: 0.XX
来源: [来源标签]
主题: 主题名称
分类: 一级分类/二级分类
tags: [标签1, 标签2, 标签3]
---
```

> ⚠️ **注意**: `入库日期` (YAML 元数据) 与**文件名日期前缀**不同：
> - **文件名日期**（`YYYY-MM-DD`）— 使用**文章/播客原始发布日期**（见 Step 7.1）
> - **YAML 中的 `入库日期`** — 记录该条目写入知识库的日期，与文件名无关
> - 两者可能不同（如 B 站视频入库晚于发布数天甚至数月），各自独立使用

**标签规范**：
- 优先使用知识库中已有的标签，不随意创建新标签
- 标签数量 3-5 个为宜
- 避免使用宽泛标签（如"AI"、"科技"），使用更具体的标签（如"大语言模型"、"Scaling Law"）

---

### Step 5: 置信度逐条标注

对每条核心结论单独标注置信度，与整体评分形成对照：

```markdown
- **AGI 可能在 5-10 年内实现** [置信度:0.75] [来源:外部-Bilibili/BVxxx]
  → 理由是 LLM 的 scaling law 仍在持续，但存在推理瓶颈
```

**规则**：
- 整体置信度沿用 Researcher 评分，不得无依据修改
- 单项置信度可在整体 ±0.10 范围内微调，但需注明理由
- 若发现某条结论明显偏离 Researcher 评分，标注差异并回退

---

### Step 6: 质量自检

精炼完成后，逐项检查：

- [ ] 核心结论是否覆盖原文最重要的 3-5 个命题？
- [ ] 每条结论是否有 `[置信度:0.XX]` 标注？
- [ ] 每条事实性陈述是否有 `[来源:XXX]`？
- [ ] 结构是否符合「核心结论→详细内容→来源」三层？
- [ ] YAML 头部是否完整（type/入库日期/置信度/来源/主题/分类/tags）？
- [ ] 标签是否使用已有标签，没有随意创建？

| 自检结果 | 下一步 |
|----------|--------|
| ✅ 全部通过 | 提交给 Writer 4重质检（如精炼者与质检者为同一 Writer，直接执行质检） |
| ❌ 未通过 | 自行修正后再次检查，最多重试 2 次；第 3 次仍不通过 → 提交 Coordinator 判断 |

---

### Step 7: 自动格式化与入库

精炼通过 Step 6 自检后，Writer 按以下流程自动入库：

#### 7.1 文件名生成
按 `YYYY-MM-DD_SourceName_Title.md` 规范生成文件名：
- `YYYY-MM-DD` — **文章/播客原始发布日期**，而非入库日期（如 2026-06-25）。从原文元数据或正文中提取：
  - **公众号文章**: 原文链接或页面顶部显示的发文日期
  - **B站视频**: 正文中 `**发布日期**:` 或 `**发布时间**:` 字段
  - **播客**: 元数据中的发布日期
  - 如无法确定原始日期，标注 `[发布日期未知]` 回退给 Researcher
- `SourceName` — 来源名称，无空格无分隔符（如 `晚点LatePost`, `opus精译`, `张小珺商业访谈录`）
- `Title` — 文章标题精简版，保留关键语义，去除冗余副标题

#### 7.2 模板组装
按 `wiki/入库模板/知识-主题.md` 模板组装最终内容：

```yaml
---
type: knowledge
入库日期: YYYY-MM-DD
置信度: 0.XX           # 沿用 Researcher 整体评分
主题: 具体主题名称
分类: 信息挖掘/<来源分类>   # 如 信息挖掘/B站
来源: [来源标签]           # 如 [外部-Bilibili/BVxxx]
tags: [tag1, tag2, ...]
---

# Title

> Summary（1-2 句概括全文）
> [来源: ...] [置信度: ...]

---

## 核心结论
（Step 2 提取的 3-5 条精炼命题，每条带 [置信度] 和 [来源]）

## 详细内容
### 背景
来源信息、发布日期、嘉宾/作者等上下文

### 章节速览（如适用）
| 时间 | 话题 | 关键内容 |
|------|------|---------|

### 关键要点
结构化要点的展开说明

### 精彩引述
> 原文引述

## 备注
精炼说明、置信度说明、来源说明

## 关联
- [[相关知识条目]]
```

#### 7.3 入库前检查
- [ ] YAML 头部完整：type / 入库日期 / 置信度 / 主题 / 分类 / 来源 / tags
- [ ] 文件名为 `YYYY-MM-DD_SourceName_Title.md` 格式（日期为原始发布日期）
- [ ] 标题行有 `> Summary` 摘要
- [ ] 内容结构包含：核心结论 / 详细内容 / 备注 / 关联
- [ ] 所有事实性陈述附带 `[来源:XXX]`，所有结论附带 `[置信度:0.XX]`

#### 7.4 完成报告
向 Coordinator 提交精炼完成报告，格式如下：
```
[Writer] 精炼完成
文件: wiki/YYYY-MM-DD_SourceName_Title.md
来源: [来源标签]
置信度: 0.XX
核心结论数: N
入库日期: YYYY-MM-DD
```

## 5. 常见问题处理

| 问题 | 处理方式 |
|------|----------|
| **原文信息密度低** | 降低精炼篇幅，只保留有价值内容；如全文无价值，标注"低密度"后退回 |
| **存在矛盾信息** | 标注矛盾点，保留两个版本，注明需要人工裁决 |
| **置信度极低的内容混入** | 回退到 Researcher 重新评估 |
| **来源缺失或多处不一致** | 回退到 Researcher 补充来源 |
| **精炼后篇幅过长（> 2000 字）** | 收紧核心结论，将次要内容移入详细内容区 |
