# 信息挖掘

> 从指定信息源采集内容，经过置信度评估、信息精炼后自动入库 Obsidian 知识库（raw → 置信度评估 → 精炼 → wiki）

---

## 1. 触发条件

### 自动触发
- **B站**：Cron 定时检查指定 UP 主的新视频
- **小宇宙**：Cron 定时拉取 RSS 更新
- **微信公众号目录**：Cron 定时扫描新链接文件

### 手动触发
- 用户直接投递内容到 `raw/`
- 用户提供公众号链接

---

## 2. 工作流

### 2.0 预检阶段（Step 0 — 来自 T007 经验）

采集任何新来源前，先执行快速预检：

- [ ] **来源类型判断**：原始内容 / 搬运 / 聚合？
- [ ] **寻求原始来源**：同内容是否存在更好的原始来源？（英文→YouTube/官网，中文→B站/公众号）
- [ ] **字幕可提取性**：
  - B站：检查 `subtitle.list` 是否为空（空→可能为硬字幕，不可 API 提取）
  - YouTube：测试 yt-dlp 能否获取字幕
  - 小宇宙：RSS shownotes 是否有内容
- [ ] **网络环境**：WebFetch / yt-dlp / Python httpx 能否访问该来源？

**预检结果 → 采集路径决策**：

| 预检结果 | 路径 | 产出位置 |
|----------|------|---------|
| API 可直接提取字幕 | A: 脚本自动采集 | `raw/` |
| 仅摘要/元数据可获取 | B: WebSearch 多源交叉验证 | `inbox/` |
| 均不可行 | C: 报告用户，建议人工提供 | — |

```
输入 → [Source Adapter] → 统一格式
     → [Researcher] → 置信度评估（Hermes 四级）
     ├─ < 0.4 → 丢弃（不浪费算力，标记后等待人工介入）
     ├─ 0.4-0.6 → process/（需双盲交叉验证 + 等人为判断）
     ├─ 0.6-0.8 → 自动触发 [Writer] 信息精炼（标注不确定点）→ wiki/
     └─ > 0.8 → 自动触发 [Writer] 信息精炼 → wiki/
     → [Coordinator] → 更新任务记录
```

### 2.1 Source Adapter（各来源转统一格式）

| 来源 | 适配方式 | 输出格式 |
|------|---------|---------|
| B站视频 | bilibili-api-python 获取字幕 | Markdown 全文 |
| 小宇宙 RSS | RSSHub 拉取 | Markdown（shownotes） |
| 微信公众号 | wechat-exporter 导出 | Markdown 全文 |
| 用户手动提交 | 直接读取 | 原格式 |

### 2.2 统一格式

所有入库内容经过 Source Adapter 后统一为：

```yaml
---
type: knowledge          # knowledge / founder
入库日期: YYYY-MM-DD
置信度: 0.00            # Researcher 填写
来源: [来源标签]        # 遵从 来源标签规范.md
主题: 主题名称
分类: 分类名
tags: []
---

## 核心结论
...

## 详细内容
...

## 来源
- [来源标签] 原文链接
```

### 2.3 置信度评估（Researcher）

遵循 `00 宪法层/置信度分级标准.md`，采用 **Hermes 7.2 四级置信度策略**：

| 置信度 | Hermes 策略 | 自动路由 | 动作 |
|--------|------------|----------|------|
| **> 80%** (0.80-1.00) | **正常流转** | 自动触发 [Writer] 精炼 → `wiki/` | 直接进入精炼环节，无需人工干预 |
| **60-80%** (0.60-0.79) | **标注不确定点** | 自动触发 [Writer] 精炼 → `wiki/` | 单 Researcher + 附带不确定点标注，精炼后入库 |
| **40-60%** (0.40-0.59) | **双盲交叉验证** | `process/` 待判断 | 路由给第二个 Researcher 独立调研，通过后再触发精炼 |
| **< 40%** (0.00-0.39) | **需人工判断** | 标记后等待人工介入 | 不浪费算力，标记后等待人工介入 |

**双盲冲突裁决**：以来源质量更高的结论为准（一手来源 > 二手来源 > 模型推断）。

**置信度降级规则**：
- 仅基于 shownotes（无逐字稿）→ 最高 0.69
- 来源未标注 → 最高 0.49
- 冲突点未处理 → 最高 0.69
- 存在未标注盲区 → 最高 0.79

### 2.4 信息精炼（Writer）

内容通过置信度评估后，自动进入**信息精炼**阶段（无需人工介入）。

> **自动触发链路**：Researcher 完成置信度评估后，Coordinator 读取评估结果：
> - 若置信度 **> 60%** → 自动将内容（含原始正文 + 置信度评分 + 评估理由）路由给 Writer
> - Writer 接收后，按 `SKILL/信息精炼.md` 执行精炼六步法 + 自动格式化入库
> - 若置信度 **40-60%** → 执行双盲交叉验证，通过后同样自动触发精炼

**输入**：通过置信度评估的内容（来自 `raw/` 或 `inbox/`）

**精炼流程**：
1. **提取核心结论**：从原文中提炼 3-5 个核心命题
2. **结构化组织**：按「核心结论→详细内容→来源」三层结构组织
3. **知识标记**：标注主题、分类、标签
4. **置信度逐条标注**：每条核心结论附带 `[置信度:0.XX]`
5. **来源引用**：所有事实性陈述附带 `[来源:XXX]`

**自动入库**：精炼完成后，Writer 按 `wiki/入库模板/知识-主题.md` 格式化，文件名按 `YYYYMMDD_Source_Title.md` 规范，自动保存到 `wiki/` 目录，无需人工确认。

---

### 2.5 输出格式

#### wiki/ 入库条目文件名规范
所有 `wiki/` 入库条目文件名遵循 `YYYYMMDD_Source_Title.md` 格式：
- `YYYYMMDD` — 入库日期（无连字符）
- `Source` — 来源名称（如 "张小珺Jùn｜商业访谈录"）
- `Title` — 文章标题精简版，保留关键语义，去除冗余
- 文件名中保留必要的特殊字符（｜、？、! 等）

示例：
```
20260625_张小珺Jùn｜商业访谈录_机器人泡沫了吗？和10亿美金创始人聊：资本轰炸下的具身智能真相.md
```

#### wiki/ 入库条目
符合 `wiki/入库模板/` 的格式，YAML 头含定量置信度。

#### process/ 待判断条目
同 wiki 格式，但附加标记：
```markdown
> **状态**: 待人为判断
> **置信度**: 0.55
> **原因**: 仅基于 RSS shownotes，缺乏完整内容
```

## 4. 质量保障

### 4.1 置信度降级规则
- 仅基于 shownotes（无逐字稿）→ 最高 0.69
- 来源未标注 → 最高 0.49
- 冲突点未处理 → 最高 0.69
- 存在未标注盲区 → 最高 0.79（Hermes < 80% 触发标注不确定点策略）

### 4.2 入库前检查（Writer 4-quality-check）
1. **来源完整性**：所有事实是否有来源标签
2. **置信度标注**：是否有显式 `[置信度:0.XX]` 标注
3. **模板合规**：是否按对应模板填写
4. **YAML 头部**：是否包含 role/created_at/version/project_id

## 5. 角色分配

| 步骤 | 角色 | 触发方式 | 职责 |
|------|------|----------|------|
| 源采集 | Coordinator/Cron | Cron 定时 | 定时触发采集任务 |
| 原始入库 | Researcher | 自动 | 标注来源、初步格式化 |
| 置信度评估 | Researcher | 自动 | 打分、写评估理由，执行 Hermes 7.2 四级策略 |
| → 路由决策 | Coordinator | 自动 | 判断置信度结果，> 60% 自动路由给 Writer |
| 信息精炼 | Writer | 自动（> 60% 触发） | 精炼六步法，提取核心结论、结构化组织、知识标记 |
| 格式化+入库 | Writer | 自动 | 按模板重排、质检、按 YYYYMMDD 规范写入 wiki/ |
| 质量稽查 | Coordinator | 抽检 | 抽检入库条目、更新项目记录 |
