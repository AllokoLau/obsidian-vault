# Coordinator 经验

## 管理类经验（跨项目可复用）

- 复杂任务先拆解为原子任务再路由，不要让一个任务跨越多个角色
- 中间材料先进 `inbox/`，最终产出进 `outputs/`
- 每次状态变更后更新项目仪表盘和协作日志
- 回退 ≥ 3 次必须升级给用户，不要自行无限重试
- 切换项目前先确认当前上下文已写入推进日志

## 注意

- 项目特定经验不存于此 → 存到 `项目空间/{项目}/`
- 临时状态不存于此 → 存到 `inbox/`

## 信息源挖掘经验（来自 T007）

### 采集前必须做预检
在开始大规模挖掘之前，先做 **Step 0 快速预检**：
1. 检查来源的字幕/文本是否可 API 提取（B站 需检查 `subtitle.list` 是否为空）
2. 检查本环境能否访问该来源（网络限制验证）
3. 如果目标是翻译/搬运内容，优先考虑原始来源（英文→YouTube，中文→B站）

### B站 硬字幕处理
B站 搬运频道可能使用**硬字幕**（文本压制在画面中），此时 B站 API 完全无法提取。遇到这种情况应立即切换路径：
- 寻找原始来源（YouTube/官网）
- 或用 WebSearch 多源交叉验证替代

### 多源交叉验证策略
当无法直接获取逐字稿时，聚合 5-6 个独立来源（Apple Podcasts + Podscripts + Snipd + 新闻 + 播客笔记站）可以提供置信度 0.70-0.85 的知识条目。搜索模板参见 `PRJ-2026-001 信息挖掘工具/inbox/T007-挖掘经验总结.md`。

### 并行处理批量条目
对于 3+ 个条目的批量处理，永远使用子 Agent 并行执行，不要串行。
