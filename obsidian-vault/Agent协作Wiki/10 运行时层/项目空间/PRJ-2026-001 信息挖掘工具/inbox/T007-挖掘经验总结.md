# T007: Dwarkesh Podcast 挖掘经验总结

> 从本次 Dwarkesh Podcast（S02）的挖掘过程中提炼的教训、有效模式和改进建议，供后续信息源挖掘加速参考。

---

## 1. 本次踩的坑

### ❌ 坑1：假设 B站搬运频道有 AI 字幕

**问题**：项目初期默认所有 B站视频都有 AI 字幕（B站已广泛部署），但 opus精译 频道上传的是**硬字幕**——翻译文本直接压制在视频画面中，API 无法提取。

**验证方法**：用 bilibili_api 检查 `video.get_info()["subtitle"]["list"]` 是否为空。本次使用 `test_subtitle.py` 验证后确认全部为空。

**代价**：前期约 30 分钟用于配置 B站 管道和验证的过程没有被浪费（QR 登录已验证、脚本已验证），但提取字幕的时间白费了。

### ❌ 坑2：默认 B站 是最佳信源

**问题**：Dwarkesh Podcast 是**英文节目**，B站 是中文搬运渠道。对于英文内容，原始 YouTube 频道才是更可靠的来源——有更完整的字幕、更及时的更新。

**教训**：对于非中文原生内容，**先评估原始来源**的可采集性，而非直接走中文搬运渠道。

### ❌ 坑3：未提前测试网络环境可访问性

**问题**：本环境有网络限制（无法直接 HTTP / DNS 访问外网），yt-dlp 和 WebFetch 都不可用。如果在规划阶段就确认这一限制，可以更早调整策略。

**教训**：首次采集一个来源前，应快速测试：
- 本环境能否访问该来源？
- 如果能，通过什么工具/API？
- 如果不能，后备方案是什么？

## 2. 本次做对的事

### ✅ 有效模式1：多源交叉验证的 WebSearch 路径

虽然无法直接访问逐字稿原文，但通过**聚合多个独立来源**获得了足够丰富的节目内容：

| 来源类型 | 提供的数据 | 可靠性 |
|----------|-----------|--------|
| Apple Podcasts 页面 | 节目标题、时长、描述 | 高（官方来源） |
| Podscripts.co、Podwise | 时间戳、话题分段 | 高（完整转录） |
| Snipd、Glasp | 精彩引述、知识摘要 | 中高（AI 提取） |
| 新闻媒体报道 | 嘉宾引述、商业分析 | 高（需交叉验证） |
| 播客笔记网站（Dexa.ai、FeiyanPod） | 标注时间戳的关键讨论点 | 中高（用户标注） |

**结论**：**5-6 个独立来源的交叉验证**可以替代逐字稿原文，产出置信度 0.70-0.85 的知识条目。

### ✅ 有效模式2：并行子 Agent 批量处理

将 9 期节目分 3 批由子 Agent 并行处理，总耗时约 4 分钟（网络搜索较快），远快于串行。

**建议**：对于 3+ 个条目的批量处理，永远使用并行子 Agent。

### ✅ 有效模式3：先做快速调研再决定采集方案

在开始大规模采集前，先让 Researcher 做**快速可行性调研**，找到频道/来源的详细信息。本次调研确认了：
- opus精译 是核心频道
- 有 9 期节目可追踪
- 但字幕类型需要验证

**代价**：约 30-40 分钟调研时间，但避免了在错误方案上浪费数小时。

## 3. 改进后的挖掘流程（V2.0）

```
用户指定来源
    ↓
[Step 0] 快速预检 ← 新增
    ├── 检查来源类型（原始 vs 搬运）
    ├── 检查字幕/文本可提取性（API 测试）
    ├── 检查网络环境可访问性
    └── 选择最优采集路径
    ↓
[Step 1] Researcher 调研
    ├── 找到具体频道/来源
    ├── 清单内容
    └── 评估采集可行性
    ↓
[Step 2] 选择采集路径（决策树）
    ├── A: API 可提取 → 脚本自动抓取 → raw/
    ├── B: 仅摘要可获取 → WebSearch 多源交叉 → inbox/
    └── C: 均不可行 → 报告用户，建议人工提供
    ↓
[Step 3] 批量处理 → 质量评估 → 入库
```

### 新增：预检清单（Step 0）

采集任何新来源前，执行以下检查：

```markdown
- [ ] 来源类型：原始内容 / 搬运内容 / 聚合内容？
- [ ] 原始来源是否存在更好的替代？(English → YouTube/官网，中文 → B站/公众号)
- [ ] 字幕/文本可提取性：
  - [ ] B站：检查 subtitle.list 是否为空
  - [ ] 小宇宙：检查 RSS shownotes 是否可用
  - [ ] 公众号：用户是否可提供链接
- [ ] 网络环境可访问性：
  - [ ] yt-dlp 能否下载？(测试单一视频)
  - [ ] WebFetch 能否访问？
  - [ ] Python httpx 能否直接请求？
```

## 4. 不同来源类型的推荐路径（更新版）

| 来源类型 | 最佳路径 | 备用路径 | 备注 |
|----------|---------|---------|------|
| B站 - AI字幕可用 | API → 字幕脚本 → raw/ | — | 已验证路径 |
| B站 - 硬字幕 | WebSearch 替代 / 用户提供文本 | 下载视频后 OCR（无此能力） | **本次新增**：需预检字幕类型 |
| YouTube 英文频道 | yt-dlp 字幕下载 → raw/ | WebSearch 多源交叉 | 需测试网络环境 |
| 小宇宙播客 | RSS shownotes → raw/ | 用户提供逐字稿链接 | 仅 shownotes 置信度 ≤0.69 |
| 微信公众号 | 用户提供链接 → 导出工具 → raw/ | — | 半自动，需用户参与 |
| PDF/网页文章 | WebFetch → raw/ | 用户提供文本 | — |

## 5. 关键词复用的加速技巧

### 可复用的搜索模板

对于播客类来源，以下搜索模式产出最快：

```
# 关键词搜索获取基本信息和时间戳
{podcast名} {嘉宾名} Apple Podcasts

# 获取时间戳和话题分段  
{podcast名} {嘉宾名} transcript timestamps

# 获取关键引述
{podcast名} {嘉宾名} quotes transcript

# 获取详细节目笔记
{podcast名} {嘉宾名} podwise / glasp / snipd
```

### 推荐的信源优先级

对于播客类知识条目：

1. **原文逐字稿**（官网/Substack）→ 置信度 0.90+
2. **YouTube 自动字幕**（yt-dlp 提取）→ 置信度 0.85-0.90
3. **Podscripts.co/Podwise 等转录站点**（WebSearch 可访问）→ 置信度 0.80-0.85
4. **WebSearch 多源交叉验证**（无原文）→ 置信度 0.65-0.85
5. **单一来源摘要** → 置信度 ≤0.65
