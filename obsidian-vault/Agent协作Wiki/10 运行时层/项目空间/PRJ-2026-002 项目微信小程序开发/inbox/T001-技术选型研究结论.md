# T001 — 技术选型研究结论

## 研究主题
为微信小程序学习项目（MVP: 商品点单→下单→结算）选择合适的技术方案。

## 核心结论

**推荐方案：微信原生开发 + 微信云开发** [置信度:0.88]

对于"学习练手"定位的单平台微信小程序项目，原生开发是最优选择。跨端框架（Taro / uni-app）的核心价值在于"跨端"，不跨则无必要承担其额外成本和风险。微信云开发提供免运维后端，适合快速验证 MVP。

## 对 Writer 的建议

### 推荐技术栈（供 Writer 撰写方案设计文档）

| 层级 | 方案 | 说明 |
|------|------|------|
| **前端框架** | 微信原生（WXML + WXSS + JS/TS） | 学习目的，不引入跨端框架 |
| **组件库** | TDesign（腾讯官方）或 Vant Weapp | 提升 UI 开发效率 |
| **后端** | 微信云开发（云数据库 + 云函数 + 存储） | 免运维，快速验证 MVP |
| **语言** | JavaScript（TypeScript 可选） | JS 降低入门门槛 |
| **工具链** | ESLint + Prettier + Git | 工程化必备 |
| **IDE** | 微信开发者工具 + VS Code | 双编辑器互补 |

### 建议的技术选型理由

1. **原生最符合学习目标**：本项目定位为"学习微信小程序开发"，原生开发直接暴露微信底层机制（App/Page/Component 生命周期、WXML 模板语法、WXSS 尺寸单位、setData 性能模型），这些知识对理解小程序本质至关重要。跨端框架会遮蔽这些内容
2. **云开发降低后端成本**：不必搭建独立服务器，云数据库 + 云函数即可支撑 MVP 需求
3. **JS 先行**：TypeScript 可在理解原生机制后再引入，不增加初期学习负担

### 结构方向
建议按以下顺序组织技术方案文档:
1. 项目架构总览
2. 前端页面结构（4 个页面：首页/商品列表 → 购物车 → 结算 → 订单）
3. 云开发数据模型（商品表、订单表、购物车表）
4. 接口设计（云函数 API）
5. 开发路线图（按功能模块拆分）

### 风险提示
- MVP 阶段暂不接入真实支付，采用模拟结算
- 商品数据可先硬编码，再迁移到云数据库
- 小程序上线后免费环境仅 15 天缓冲期 — 学习项目保持在开发版/体验版即可
