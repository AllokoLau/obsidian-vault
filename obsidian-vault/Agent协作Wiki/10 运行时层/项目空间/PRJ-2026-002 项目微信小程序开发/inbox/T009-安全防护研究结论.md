# 研究结论：微信小程序全链路安全防护

> **研究范围**: 客户端安全、云函数安全、数据库安全规则三大方向
> **适用项目**: PRJ-2026-002 微信小程序 MVP（点单→下单→结算）
> **当前基线**: 客户端使用 localStorage mock 数据，云函数已编写但未激活

---

## 一、研究摘要

微信小程序的安全防护可分为三个独立但相互配合的层级。本项目为学习型 MVP（模拟结算，不接入真实支付），安全措施应覆盖核心业务链路，但无需企业级复杂度。

| 层级 | 核心风险 | 应对策略 |
|------|---------|---------|
| **客户端** | XSS、客户端价格篡改、敏感数据泄露 | WXML 自动转义 + 云函数定价 + 最小存储 |
| **云函数** | NoSQL注入、水平越权、超卖、重复提交 | 参数白名单 + _openid 校验 + 事务 + 幂等 |
| **数据库规则** | 前端直读他人数据、越权写 | 安全规则绑死 `_openid == auth.openid` |

---

## 二、R1 客户端安全 [置信度:0.85]

### 2.1 XSS 防护

**关键发现**: 微信小程序 WXML 默认对数据绑定进行 HTML 实体转义，天然防 XSS [来源:微信官方文档/模板语法]。以下写法**天然安全**：
```wxml
<view>{{ userInput }}</view>  <!-- 自动转义 < > & " ' -->
```
需要使用 `wxParse` 或 `<rich-text>` 渲染 HTML 时才需关注 XSS，但本项目不使用富文本渲染用户输入。

**需要关注的**: 本项目中仅商品 `description` 使用富文本，数据来源于云数据库（管理员写入），不来自用户输入，风险可接受。

**结论**: 本项目客户端 XSS 风险低，无需额外措施。

### 2.2 客户端价格防篡改 [置信度:0.90]

**核心原则**: **不信任客户端传递的价格** [来源:CSDN-微信小程序开发全攻略/安全误区]。

当前问题：
- `checkout.js` 用 localStorage 中存储的 `item.price` 计算金额
- 用户可通过修改本地存储或拦截请求篡改价格

解决方案：
- 下单时客户端**只传 `goodsId` + `quantity`**，由云函数从数据库读取当前价格
- 客户端仅作展示用途，下单金额以服务端计算为准

**盲区**: 商品价格在数据库被修改后，待支付订单使用的价格可能与用户下单时看到的价格不一致。学习项目可接受，生产环境需在创建订单时锁定价格到订单快照。

### 2.3 本地存储安全 [置信度:0.85]

敏感数据（如 token、用户信息）不应明文存储于 `wx.getStorageSync` [来源:微信官方文档/数据安全]。

本项目当前使用 localStorage 存储：
- ✅ 商品列表（公开数据，风险低）
- ✅ 购物车（用户自有数据，仅展示用途）
- ❌ **订单数据** - 当前在客户端生成并存 localStorage，必须迁移至云数据库
- ⚠️ `userInfo`（头像+昵称） - 非敏感，但建议仅展示不持久化

### 2.4 客户端安全建议总结

| 措施 | 优先级 | 说明 |
|------|--------|------|
| 订单操作全部走云函数 | **P0** | 消除客户端生成订单的安全风险 |
| 下单时客户端只传 goodsId+quantity | **P0** | 服务端定价，消除价格篡改 |
| localStorage 仅存购物车等非敏感数据 | P1 | 订单数据迁至云数据库 |
| 敏感操作二次确认弹窗 | P1 | 取消订单已实现，结算前再加一次确认 |

## 三、R2 云函数安全 [置信度:0.88]

### 3.1 NoSQL 注入防护 [置信度:0.92]

**关键发现**: 云数据库底层兼容 MongoDB 协议，支持 `$ne`、`$gt`、`$in`、`$or` 等运算符。如果将前端传参直接拼入 `.where()`，可被攻击 [来源:CSDN-云数据库NoSQL注入深度剖析]。

**危险写法（禁止）**：
```javascript
// 直接将前端参数作为查询条件
db.collection('user').where(event.query).get()
```

**安全写法**：
```javascript
// 白名单提取字段
const { status } = event
const query = { _openid: wxContext.OPENID }
if (['pending','confirmed','completed','cancelled'].includes(status)) {
  query.status = status
}
db.collection('orders').where(query).get()
```

当前代码审查：`getOrders` 和 `updateOrderStatus` 已采用白名单方式，**未发现 NoSQL 注入风险** ✅

### 3.2 参数校验规范 [置信度:0.90]

所有云函数入口必须执行 [来源:CSDN-云数据库安全深度剖析]：

```javascript
exports.main = async (event, context) => {
  // 1. 类型检查
  if (typeof event.param !== 'string') return { code: -1, msg: '参数类型错误' }
  // 2. 长度限制
  if (event.param.length > 100) return { code: -1, msg: '参数过长' }
  // 3. 白名单枚举
  const ALLOWED = ['a', 'b']
  if (!ALLOWED.includes(event.param)) return { code: -1, msg: '无效参数' }
  // 4. 格式正则
  if (!/^[a-f\d]{24}$/.test(event.id)) return { code: -1, msg: 'ID格式错误' }
}
```

当前问题：现有云函数的参数校验**过于宽松**，只检查存在性，未做类型、长度、格式校验。

| 云函数 | 现有校验 | 需要补充 |
|--------|---------|---------|
| createOrder | `if (!items)` | 类型+每个 item 的字段校验+数量范围 |
| getOrders | 无 | status 白名单+分页参数校验 |
| updateOrderStatus | 长度+白名单 | orderNo 格式校验+状态机约束 |
| initGoods | 无 | 管理员权限校验 |

### 3.3 事务处理 [置信度:0.85]

**关键发现**: 微信云开发数据库支持 `startTransaction()`，用于保证多个写操作的原子性 [来源:微信官方文档/云数据库/事务]。

```javascript
const transaction = await db.startTransaction()
try {
  await transaction.collection('orders').add({ data: orderData })
  await transaction.collection('goods').doc(id).update({
    data: { stock: _.inc(-quantity) }
  })
  await transaction.commit()
} catch (err) {
  await transaction.rollback()
  throw err
}
```

**createOrder 必须使用事务**：订单写入和库存扣减是原子操作，否则高并发下会出现"下单成功但库存未扣减"或"库存扣减但下单失败"。

当前问题：createOrder 的库存扣减和订单写入是**两个独立操作**，无事务保护。

### 3.4 权限校验 [置信度:0.90]

**OpenID 校验**: 云函数中通过 `cloud.getWXContext().OPENID` 获取用户身份 [来源:微信官方文档/云函数/鉴权]。所有数据操作必须限定 `_openid == wxContext.OPENID`。

当前代码审查：
- `createOrder` ✅ 获取了 `_openid` 但未在订单中显式校验前端传入的 items 是否属于当前用户（当前使用购物车，但 items 来自前端）
- `getOrders` ✅ 已限定 `_openid`
- `updateOrderStatus` ✅ 已限定 `_openid`
- `initGoods` ❌ **无任何权限控制，任意用户可调用**

**频率限制**: 云函数免费版有配额限制（20万次/月），但微信云开发**未提供内置频率限制** [来源:微信开放社区]。学习项目中通过客户端节流控制即可，无需复杂方案。

### 3.5 云函数安全建议总结

| 措施 | 优先级 | 影响范围 |
|------|--------|---------|
| createOrder 启用事务（订单写入+库存扣减） | **P0** | 核心数据完整性 |
| createOrder 服务端定价 | **P0** | 消除价格篡改 |
| 所有云函数加强参数校验（类型+长度+格式） | **P0** | 全部云函数 |
| initGoods 添加管理员权限校验 | **P0** | 数据安全 |
| updateOrderStatus 添加状态机约束 | P1 | 订单状态一致性 |
| getOrders 添加分页限制 | P1 | 性能与配额 |

## 四、R3 数据库安全规则 [置信度:0.90]

### 4.1 安全规则概述

微信云开发数据库可设置 JSON 格式的**安全规则**，比基础的四种权限模式更精细 [来源:官方文档-微信云开发数据库安全规则]。常用变量：

| 变量 | 含义 |
|------|------|
| `auth.openid` | 当前用户的 openid |
| `doc._openid` | 文档的创建者 openid（云开发自动注入） |
| `now` | 当前时间戳 |

### 4.2 本项目需要的安全规则

**`goods` 集合** — 商品目录，所有人可读，仅管理员（云函数）可写：
```json
{
  "read": true,
  "write": false
}
```
说明：商品数据为公开目录。写操作仅通过云函数 `initGoods` 进行（云函数不受规则限制）。

**`cart` 集合** — 购物车，仅本人可读写：
```json
{
  "read": "doc._openid == auth.openid",
  "write": "doc._openid == auth.openid"
}
```
说明：每个用户只能访问自己的购物车数据。

**`orders` 集合** — 订单，仅本人可读，写操作仅通过云函数：
```json
{
  "read": "doc._openid == auth.openid",
  "write": false
}
```
说明：订单只能通过云函数创建/更新，前端不可直接写入。这是最严格的配置。

### 4.3 注意事项

1. **查询条件必须匹配安全规则** [置信度:0.90]：前端 `where()` 的查询条件必须能通过安全规则检查。例如 `orders` 规则为 `doc._openid == auth.openid`，则前端查询必须携带 `where({ _openid: '{openid}' })`。
2. **安全规则升级兼容**：从基础权限切换到自定义规则后，某些 doc 操作需转为 where 操作。
3. **云函数特权**：云函数内操作不受安全规则限制，所以关键写操作必须通过云函数 + 自校验 [置信度:0.95]。
4. **开销考虑**：`get()` 函数消耗配额，本项目暂不需要跨集合查询。

## 五、盲区与局限

| 盲区 | 说明 | 影响 |
|------|------|------|
| 微信云开发事务超时限制 | 未实测交易超时时间和并发下行为 | 学习项目模拟结算，无真实支付并发，低影响 |
| admin 权限授予方式 | 微信云开发没有内置 admin 角色，需自定义实现 | initGoods 的 admin 校验需简单实现（硬编码管理员 openid） |
| 云函数冷启动对安全的影响 | 冷启动时云函数可能更慢，但不影响校验逻辑 | 无需额外处理 |
| 微信安全检测模板 | 有内容安全 API（文本/图片检测）但本项目不涉及 UGC | 未纳入研究范围 |

## 六、结论与建议

针对本项目的全链路安全防护，建议实施以下措施（按优先级排列）：

**P0（核心业务安全，必须做）**：
1. ✅ createOrder 启用事务
2. ✅ createOrder 服务端定价（仅接收 goodsId+quantity）
3. ✅ 所有云函数加强参数校验
4. ✅ initGoods 添加管理员权限校验
5. ✅ 设置数据库安全规则（orders 写权限仅云函数）

**P1（增强安全，建议做）**：
6. ✅ updateOrderStatus 状态机约束
7. ✅ getOrders 分页+字段限制
8. ✅ 客户端结算操作二次确认

**P2（学习体验，可选做）**：
9. ⌛ 云函数错误日志分级
10. ⌛ 客户端请求节流（util.js已有 throttle 函数）

> **整体置信度: 0.85** — 主要来源为微信官方文档和社区最佳实践，未在真实生产环境下验证。作为学习项目，覆盖核心安全面即可。
