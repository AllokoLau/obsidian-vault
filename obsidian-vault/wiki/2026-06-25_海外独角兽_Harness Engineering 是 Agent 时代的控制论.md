# Harness Engineering 为什么是 Agent 时代的"控制论"？

> **来源**: [外部-微信公众号/Harness Engineering 为什么是 Agent 时代的"控制论"？]
> **公众号**: 海外独角兽
> **原文链接**: https://mp.weixin.qq.com/s/SVUybMZb6uh5OCR3ceoBVA
> **入库日期**: 2026-06-25

---

## 核心摘要

1. Harness Engineering 的本质是控制论（Cybernetics）在现代软件工程中的再次体现——工程师从"亲手操作系统"转变为"设计让系统自动运转的机制"。这个模式历史上已出现三次：瓦特离心调速器（蒸汽机）、Kubernetes（容器编排）、以及当前的 Agent Harness。
2. LLM 同时改变了代码质量判断和代码修改两端，第一次让反馈回路可能在架构决策层面闭合，这是 Agent 能够自主工作的根本前提。
3. Agent 失败的原因往往不是能力不够，而是"什么是对的"这一判断标准没有从人的大脑中显式写出——这些知识锁在人脑中，Agent 无法自主学习进化。
4. 真正的工程实践（文档、自动化测试、编码成规则的架构决策、快速的反馈回路）从未改变，但 Agentic Engineering 让跳过这些实践的代价变得极端——不做这些事的成本已无法承受。
5. 生成与验证之间存在不对称性（P vs NP 的核心直觉）：验证一个答案比生成一个正确答案更容易，因此工程师的新角色是定义"什么是正确"，而不是比机器写得快。

## 正文

### 同一个模式出现了三次

作者 George Zhang（OpenClaw 维护者）指出，Harness Engineering 不是新鲜事物，而是控制论在软件工程中的再次体现。

- **第一次（18 世纪 80 年代）**：詹姆斯·瓦特发明离心调速器，工人从亲手拧阀门变成设计调速器本身。
- **第二次（Kubernetes 时代）**：工程师从手动重启服务变成编写系统需要对齐的目标 spec，控制器自动监测和修正偏差。
- **第三次（现在）**：OpenAI 描述的 Harness Engineering——工程师不再亲手写代码，而是设计运行环境、构建反馈回路、把架构约束转化成可执行的规则。

诺伯特·维纳在 1948 年将此命名为"控制论"（cybernetics），来自希腊语 κυβερνήτης（舵手），Kubernetes 也来自同一词根。本质含义一致：你不再需要亲手拧阀门，而是开始掌舵。

### 为什么代码库是最后被攻克的领域

代码库并非没有反馈回路，只是这些回路存在于底层：编译器在语法层面闭合回路，测试套件在行为层面闭合回路，Linter 在代码风格层面闭合回路。但再往上一层——"这个改动符合系统的整体架构吗？""这个技术方案是对的吗？"——没有任何自动化机制。

LLM 同时改变了判断端和执行端：它能像人一样判断代码质量和进行改动。这是第一次，反馈回路可能在真正关键的决策层面闭合。

### 如何校准传感器和执行器

让 Agent 能够执行测试、CI 产出可解析结果、报错信息指向具体修复方向，只是最低门槛。真正困难的问题在于：如何基于对系统的认知来校准这套传感器和执行器。

Carlini 的演示（16 个 agent 并行协作构建 C 编译器）中，他大部分精力花在设计测试、运行环境和反馈机制上。大多数人的失败归于 Agent 本身，但真正的原因是"什么是对的"这一知识锁在人脑中从未被写出来。

Agent 不会自主学习进化。如果不把这些知识写下来，Agent 第一百次犯的错会和第一次一模一样。真正需要做的工作是：把判断标准变成机器可读的形式——架构文档、自定义 Linter、黄金原则。

OpenAI 曾每周花 20% 时间清理"AI slop"，后来把标准直接编进 Harness 本身，问题才得到根本解决。

### 唯一的出路

文档、自动化测试、编码成规则的架构决策、快速的反馈回路——这些本来就是正确的工程实践。过去三十年每一本软件工程的书都在推荐这些，但大多数人选择跳过，因为跳过的代价来得很慢。

Agentic Engineering 让这个代价变得极端：
- 跳过文档，Agent 会在每一个 PR 上以机器速度全时段重复同样的错误。
- 跳过测试，反馈回路无法闭合。
- 跳过架构约束，代码漂移速度远快于手动修复。
- 如果 Agent 不知道什么叫做整洁的代码，就无法用 Agent 收拾烂摊子。

未经校准的 Agent，制造了问题也解决不了问题。

### P vs NP 的启示

生成答案和验证答案之间存在明显的不对称性：验证一个答案比生成一个正确答案要容易得多（Cobbe 等研究者已在 LLM 上验证）。这个不对称性给出了明确方向：不必在编写代码上比机器更快，只要能高效评估它的产出——定义什么是"正确"，识别输出偏离目标的地方，判断整体方向是否正确。
