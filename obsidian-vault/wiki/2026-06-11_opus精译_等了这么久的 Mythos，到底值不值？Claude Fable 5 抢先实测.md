### 关键要点

1. **Fable 5 的定位与定价**：作为 Anthropic 第三层智能（Sonnet → Opus → Mythos）的首个模型，Fable 5 的定价显著高于 Opus（输入 5 倍、输出约 2.5 倍），且 token 消耗量约为其他模型的 2 倍，实际使用成本极高。评测者全程使用"极高"配置测试以避免评论质疑。

2. **"资深工程师悖论"**：Fable 5 的行为被类比为资深工程师——会彻底调查所有细节、考虑 120% 的交付质量。但这意味着它不适合需要快速迭代、轻度触及的产品开发场景。有时需要的不是最聪明而是"稍微笨拙一些"的工程师。

3. **视觉能力的突破**：评测者明确表示在视觉领域看到了"突破"，特别是在文档排版、PDF 解析、手写文档布局等方面。这被认为是 Fable 5 区别于 Opus 最明显的进步领域。

4. **文本输出的可用性危机**：产品规格和策略类写作任务中，Fable 5 输出过于密集、内部引用过多、缺乏宏观结构，极难阅读和理解。评测者明确建议用 Sonnet 或 Opus 处理这些任务。

5. **安全机制的创新设计**：最大的安全创新是"回退而非拒绝"——触发安全分类器后自动切换到 Opus 4.8 而非直接中断请求。这意味着用户不会完全失去服务，只是降级到较低智能层级。30 天保留策略仅用于监控不当使用，不用于训练。

6. **分层使用策略（Advisor Strategy）**：新推出的顾问策略将 Fable 5 作为资深顾问，用 Sonnet 或 Opus 作为执行层。事实上，已有用户在 API 中用 Opus + Sonnet 组合实践了这一模式。

7. **多智能体编排的早期状态**：虽然官方宣传了子代理启动、动态工作流、多日会话等能力，但实测中约 3 小时后子代理停滞，评测者认为这更可能是 Claude Code 平台的成熟度问题而非模型推理能力问题。

8. **评测者对 Fable 5 的总体态度**：不讨厌这个模型，明确认为它在技术栈中"肯定有位置"，但强调需要精细化地决定何时使用——"模型生态系统中各有其用"。
