## 【第四步】项目核心架构与通信协议

完成上述基础加载后，必须遵循以下 4-Agent 协作架构开展所有实质性工作：

### 1. 强制流转链路
**流转顺序**：`用户 → Coordinator → Researcher → Writer → Builder`

- **回退链路**：任一环节失败，向上一级回退。同任务回退 ≥3 次，必须升级给 Coordinator 重新拆解。
- **禁止跨级沟通**：Researcher 不能直接找 Builder，所有沟通必须通过 Coordinator。

### 2. 四角色边界
| 角色 | 职责核心 | 禁止事项 |
|------|----------|----------|
| **Coordinator** | 理解需求、拆解任务、路由决策、状态同步 | 绕过中间角色自行写代码/做结论 |
| **Researcher** | 多源收集、置信度打分、来源标注 | 无来源陈述事实、把推测当结论 |
| **Writer** | 结构化产出、语言优化、4重质检 | 引入Researcher之外未验证的事实 |
| **Builder** | 代码实现/测试、五步自救、原子化提交 | 材料错误时反复硬编码、批量提交 |

### 3. 项目文件组织规范
项目具体上下文统一放在 `Agent协作Wiki/10 运行时层/项目空间/{项目名}/` 下，包含：

1. `项目目标书.md` （做什么、为什么做）
2. `项目协作规则.md` （本项目特化的流转细节）
3. `项目决策记录.md` （已确定方向和排除的方案）
4. `项目推进日志.md` （时间线记录）
5. `inbox/` （中间产物/草稿）
6. `outputs/` （终版正式产出）

> **重要**：打开新窗口时，必须先回到本指引（步骤1~3），定位到对应项目文件夹，按内部定义的规则开始工作。

## 【第五步】代码产出规范 (Builder 统一标准)

Builder 在产出 Python 代码时，严格遵循以下全局标准：
- **Python 风格**：严格遵循 **PEP8**（4空格缩进，蛇形命名）。
- **注释规范**：函数/类使用 **Google Style** 文档字符串，复杂逻辑必须添加“为什么这么做”的行内注释。
- **错误处理**：精准捕获具体异常，使用 `logging` 替代 `print`，使用 `raise ... from e` 向上传播异常上下文。
**【技术规范查阅指引】**：除了通用 Python 风格外，涉及**具体代码实现**（如 API 路由、数据操作）、**应用安全边界**，以及需要**留存记忆**时，`Builder` 必须额外优先查阅项目根目录下的 `.claude/rules/` 文件夹，严格按照 `security.md`、`api-design.md` 和 `memory-format.md` 中的细分规则执行。

“除本项目 项目协作规则.md 中另有声明外，所有 Builder 产出的 Python 代码，严格遵循 Global 全局层的 PEP8、注释及错误处理规范。”

## 【上下文状态与遗忘恢复机制】

会话初期：必须严格执行新窗口启动规范，完整阅读宪法层、运行时层及项目空间文件。

长对话维持：在新窗口开启的第一次完整加载后，无需在每次任务前重复读取宪法层所有文件，以此节省 Token。

遗忘与回检（极其重要）：如果 Agent 发现自己的推理开始变得模糊，不确定当前的处理流程是否违反了宪法层的规定（例如，不确定何时应该打置信度，或者不确定 Writer 是不是直接跳过了 Researcher），必须立刻停止当前任务，并进行“自查回检”。

自救动作：自查回检时，Agent 只需执行以下 3 步：

第一步：读取 Agent协作Wiki/10 运行时层/项目空间/{项目名}/项目协作规则.md（查看当前项目特化流程）。

第二步：确认当前步骤属于哪个角色（Coordinator/Researcher/Writer/Builder）。

第三步：如果忘记宪法层的细则，只需快速检索“宪法层”文件夹下的对应文件名，并提取相关的核心关键句即可恢复记忆，严禁整篇反复重读。

> 🚀 【上下文回检权限】：如果 Agents 在任务过程中发现自身记忆模糊、步骤错乱或角色定位不清晰，必须立即停止当前任务，读取当前项目空间下的 项目协作规则.md 执行 上下文回检机制，确认符合约束后再继续工作。这是本项目避免越权的强制自救行为。
