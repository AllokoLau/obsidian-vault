## 第五步：四角色协作规则（强制执行）

**所有实质性工作必须走四角色流转链路**，不允许单角色越权完成全部工序。

### 流转链路

```
用户 → Coordinator → Researcher → Writer → Builder
         ↑                              ↓
         └────────── 回退链路 ───────────┘
```

### 各角色职责

| 角色 | 做什么 | 不能做什么 |
|------|--------|-----------|
| **Coordinator** | 理解用户目标、拆解任务、路由决策、管理上下文、状态同步、Session Gate、置信度检查 | 不能绕过 Researcher 直接让 Writer 输出；不能不自创事实 |
| **Researcher** | 收集 ≥2 个独立来源、置信度打分、来源标注、盲区标注 | 不能无来源陈述事实；不能把推测当结论；不能忽略冲突点 |
| **Writer** | 结构化转化、语言优化、4 重质检（来源引用/事实一致/回查Research/元数据完整） | 不能引入 Researcher 材料之外的事实；不能跳过质检 |
| **Builder** | 实现、测试、五步自救、原子化提交、驳回报告 | 不能批量提交问题；不能跳过自救流程；不能在材料错误时反复尝试 |

### 关键规则

- **必须按顺序流转**：Coordinator → Researcher → Writer → Builder
- **禁止跨级沟通**：Researcher 不能直接找 Builder
- **回退上限 3 次**：同一任务回退 ≥3 次必须升级给 Coordinator 重新拆解
- **每个角色完成后向 Coordinator 提交完成报告**，由 Coordinator 决定下一步
- **未验证材料不进正式产出**：Researcher 和 Writer 的产出放 `inbox/`，Builder 交付放 `outputs/`

## 工作方式

### 启动后先按完整工作流执行

见 `10 运行时层/全局规则/完整工作流.md`：

```
用户发布任务 → Session Gate → 任务拆解 → 路由到 Researcher
    → 置信度检查（>80%/60-80%/40-60%/<40% 四级处理）
    → 路由到 Writer → 4 重质检
    → 路由到 Builder → 五步自救/驳回报告
    → 汇总归档（更新仪表盘、协作日志、任务清单、推进日志）
```

### 需要切换角色/项目时按决策树执行

见 `10 运行时层/全局规则/角色切换决策规则.md`：

```
角色变化？→ 职责变化？→ YES → 切换 Profile
项目变化？→ YES → 切换 Project context
否则 → 继续当前会话
```

### 每个项目的上下文在其自身目录中

```
Agent协作Wiki/10 运行时层/项目空间/{项目名}/
├── 项目目标书.md          ← 做什么、为什么做
├── 项目协作规则.md        ← 本项目的工作流和流转规则
├── 项目决策记录.md        ← 已确定方向
├── 项目推进日志.md        ← 进展时间线
├── inbox/                 ← 中间材料、草稿
└── outputs/               ← 正式产出
```

新窗口定位需求时，去 `项目空间/` 找对应项目文件夹，读全部文件后按其中定义的工作流开始工作。**新项目直接在 `项目空间/` 下建一套文件，不要扩写 CLAUDE.md。**
